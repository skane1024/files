### **`OpDynamicRule` 含有 `union` 和枚举，本身是 POD，但 `std::string` 不是，需要自定义二进制序列化协议**

`std::pair<std::string, OpDynamicRule>` 中的 `std::string` 是堆上的动态对象，不能直接 `fwrite` 整个 `vector`。需要把字符串的长度和内容分开写入，`OpDynamicRule` 部分才可以整块 `memcpy`。

## **一、二进制文件格式设计**

设计一个简单紧凑的格式，逐条记录每个算子：

```
[ 文件头 ]
  uint32_t  count          // 算子总数

[ 每条记录，重复 count 次 ]
  uint32_t  name_len       // 算子名字符串长度（不含 '\0'）
  char[]    name_data      // 算子名原始字节
  OpDynamicRule rule_data  // 固定大小的规则结构体，直接二进制写入
```

这样读写双方只要约定好这个格式，就能做到零解析开销地还原数据。

## **二、写入实现**

```cpp
#include <cstdint>
#include <cstring>
#include <fstream>
#include <stdexcept>
#include <string>
#include <unordered_map>

// 写入辅助：向文件写入固定字节
template <typename T>
static void writeRaw(std::ofstream& ofs, const T& val) {
    ofs.write(reinterpret_cast<const char*>(&val), sizeof(T));
}

bool saveToBinary(const std::string& filepath,
                  const std::unordered_map<std::string, OpDynamicRule>& rules)
{
    std::ofstream ofs(filepath, std::ios::binary);
    if (!ofs) return false;

    // 1. 写入算子总数
    uint32_t count = static_cast<uint32_t>(rules.size());
    writeRaw(ofs, count);

    // 2. 逐条写入
    for (const auto& kv : rules) {
        // 2a. 写入名称长度 + 名称内容
        uint32_t name_len = static_cast<uint32_t>(kv.first.size());
        writeRaw(ofs, name_len);
        ofs.write(kv.first.data(), name_len);

        // 2b. 写入规则结构体（整块 POD，直接写）
        writeRaw(ofs, kv.second);
    }

    return ofs.good();
}
```

## **三、读取实现**

```cpp
// 读取辅助：从文件读出固定字节
template <typename T>
static void readRaw(std::ifstream& ifs, T& val) {
    ifs.read(reinterpret_cast<char*>(&val), sizeof(T));
}

bool loadFromBinary(const std::string& filepath,
                    std::unordered_map<std::string, OpDynamicRule>& rules)
{
    std::ifstream ifs(filepath, std::ios::binary);
    if (!ifs) return false;

    // 1. 读取算子总数
    uint32_t count = 0;
    readRaw(ifs, count);

    rules.reserve(count); // 预分配桶，减少 rehash

    // 2. 逐条读取
    for (uint32_t i = 0; i < count; ++i) {
        // 2a. 读取名称
        uint32_t name_len = 0;
        readRaw(ifs, name_len);

        std::string name(name_len, '\0');
        ifs.read(&name[0], name_len);

        // 2b. 读取规则结构体
        OpDynamicRule rule;
        readRaw(ifs, rule);

        rules[std::move(name)] = rule;
    }

    return ifs.good();
}
```

        assert(it->second.data.matmul.block_index == 2);
    }
}

## **五、几个需要注意的坑**

**结构体对齐与填充（Padding）问题**是最容易踩的坑。编译器可能在结构体成员之间插入填充字节以对齐内存，这部分字节的内容是未定义的。直接 `fwrite` 时这些字节也会被写入文件，只要读写双方使用**同一个编译器、同一套编译选项**，就不会有问题。如果需要跨平台或跨编译器，可以加上 `#pragma pack(1)` 或 `__attribute__((packed))` 消除填充，但这会带来轻微的访问性能损失。

**大小端（Endianness）问题**在同一架构的机器上不需要关心。如果文件需要在 x86 和某些嵌入式 ARM 大端平台之间交换，需要在写入/读取时做字节序转换（`htonl` / `ntohl`）。

**`union` 的未使用部分**同样存在未定义字节。例如一个 `SOFTMAX` 类型的 `OpDynamicRule`，其 `union` 中只有 `generic_shape` 被初始化，`matmul` 和 `copy` 的剩余字节是脏数据，但这不影响正确性，因为读取时只会按 `type` 访问对应的分支。

**版本兼容性**：如果将来结构体定义发生变化（增删字段），旧的 `.bin` 文件将无法正确读取。建议在文件头加一个 `uint32_t version` 字段，方便后续做版本检测和迁移。

```cpp
// 推荐的文件头结构
struct BinFileHeader {
    uint32_t magic   = 0x4F505255; // "OPRU" 魔数，用于校验文件格式
    uint32_t version = 1;          // 格式版本号
    uint32_t count   = 0;          // 算子总数
};
```

加上这个文件头后，读取时先校验 `magic` 和 `version`，可以有效防止读取错误格式的文件导致的崩溃。
