### **第2页：问题背景**

在 ONNX 模型推理过程中，除了模型权重和输入输出之外，还需要保存大量中间计算结果。  
这些中间结果通常存放在临时缓冲区，也就是 temp buffer 中。

对于存在残差连接、分支结构、Concat 或多路汇聚的网络，一些前面算子的输出不能立即释放，而必须保留到后续算子使用时。  
因此，推理过程中的 temp buffer 峰值往往成为内存受限设备上的关键瓶颈。

**核心问题：**

- 一个静态 shape 模型最少需要多少 temp buffer 内存
- 如何在不改模型功能的前提下降低推理过程中的 temp buffer 峰值

---

### **第3页：什么是 Temp Buffer**

Temp buffer 指模型推理过程中，为保存中间张量而占用的临时内存。

它通常不包括：

- 模型权重
- 外部输入输出缓冲
- 持久参数区

它主要包括：

- 算子输出后、在后续节点使用前必须保留的中间结果
- 部分算子执行过程中产生的临时工作区

可以强调一句：

**本次讨论重点是中间激活张量的临时存储，而不是权重内存。**

---

### **第4页：为什么 Skip Connection 会增加内存**

Skip Connection 的本质是：  
前面层产生的输出，需要跨过若干中间层，在后面再次参与计算。

以残差结构为例：

$$
y = F(x) + x
$$

这里的 $x$ 不能在进入分支计算后立即释放，而必须一直保留到最后与 $F(x)$ 相加。  
因此在分支内部不断生成新中间张量的同时，旧张量 $x$ 仍然占据内存。

这会导致：

- 中间张量生命周期变长
- 同时存活的张量数增多
- temp buffer 峰值上升

### **第5页：静态 Shape 下最小 Temp Buffer 的定义**

对于静态 shape 模型，每个中间 tensor 的大小是已知的，因此可以精确分析 temp buffer 的需求。

一个中间张量有三个关键属性：

- 产生时刻
- 最后使用时刻
- 张量大小

在执行过程中，凡是“已经产生但尚未完成最后一次使用”的张量，都必须保留在 temp buffer 中。

因此，最小 temp buffer 可以定义为：

**整个推理时间线上，所有同时存活的中间张量所占内存的最大值。**

也就是中间激活内存的峰值。

---

### **第6页：数学描述**

设某个张量 $t$ 的属性为：

- $birth(t)$：产生时刻
- $death(t)$：最后一次使用时刻
- $size(t)$：内存大小

那么在执行时刻 $k$，所有存活张量的集合为：

$$
L(k) = \{t \mid birth(t) \le k < death(t)\}
$$

该时刻所需 temp buffer 为：

$$
M(k) = \sum_{t \in L(k)} size(t)
$$

整个推理过程中的最小需求峰值为：

$$
M_{peak} = \max_k M(k)
$$

这一数值就是固定执行顺序下的理论最小 temp buffer 下界。

### **第7页：一个直观例子**

假设模型片段如下：

- $t_1 = Op1(x)$
- $t_2 = Op2(t_1)$
- $t_3 = Op3(t_1)$
- $t_4 = Add(t_2, t_3)$

设：

- $t_1 = 10$ MB
- $t_2 = 12$ MB
- $t_3 = 12$ MB
- $t_4 = 12$ MB

执行过程中：

- 生成 $t_1$ 后，占用 10 MB
- 生成 $t_2$ 后，$t_1 + t_2 = 22$ MB
- 生成 $t_3$ 后，$t_1 + t_2 + t_3 = 34$ MB
- 执行 Add 后，旧张量可释放

因此峰值 temp buffer 为：

**34 MB**

这个例子说明：  
一旦存在分支复用，早期结果不能及时释放，峰值就会上升。

### **第8页：影响 Temp Buffer 的主要因素**

静态 shape 下，temp buffer 的大小主要取决于以下因素：

- 网络拓扑结构  
  分支、残差、Concat、U-Net 跳连都会增加生命周期重叠

- 中间张量大小  
  分辨率越高、通道越多，单个激活越大

- 执行顺序  
  不同拓扑序会影响张量是否长时间同时存活

- 数据类型  
  FP32、FP16、INT8 的内存占用不同

- 算子是否支持原地执行  
  原地执行可以减少新增 buffer

- 是否进行算子融合  
  融合后可减少中间结果落地

### **第9页：最基础的优化思路——生命周期分析**

降低 temp buffer 的第一步，是分析每个 tensor 的生命周期。

需要知道：

- tensor 在哪个节点产生
- tensor 被哪些节点使用
- 最后一次使用发生在哪一步

一旦某个 tensor 已经完成最后一次使用，就应当立即释放其内存，供后续张量复用。

因此，生命周期分析的目标是：

**让每一块内存尽可能早释放，尽可能快复用。**

---

### **第10页：关键优化一——内存复用**

如果两个 tensor 的生命周期不重叠，那么它们可以共用同一块内存。

例如：

- tensor A 在第 10 步后失效
- tensor B 在第 11 步才产生

那么 A 和 B 可以映射到同一个 buffer。

这类优化的典型实现方式是：

- 预先申请一块大的内存池
- 按 tensor 生命周期进行静态分配
- 生命周期结束后回收到空闲块池
- 后续张量优先复用已有空间

收益：

- 显著降低 temp buffer 峰值
- 避免频繁 malloc/free
- 减少内存碎片

---

### **第11页：关键优化二——优化执行顺序**

在不违反依赖关系的前提下，不同执行顺序会影响峰值内存。

优化原则是：

- 尽快消费大张量
- 尽快释放不再使用的中间结果
- 避免多个大分支结果长期同时存活

也就是说，调度策略应尽量减少 live tensor 的数量与重叠时间。

对于有多个并行分支的图结构，这种优化尤其有效。

---

### **第12页：关键优化三——原地执行**

某些算子在满足条件时，可以直接覆盖输入张量，而不再额外申请输出 buffer。

例如某些逐元素算子可能支持原地执行：

- ReLU
- Clip
- 部分 Unary 算子

原地执行的前提是：

- 输入 tensor 后续不再被其他节点使用
- 输出 shape 与输入一致
- 算子语义允许覆盖输入

收益是：

**减少额外中间张量分配，从而降低 temp buffer 需求。**

---

### **第13页：关键优化四——算子融合**

如果多个相邻算子可以融合成一个执行单元，中间结果就不需要单独落地到内存。

例如：

- Conv + Bias + ReLU
- MatMul + Add
- 一些归一化与激活组合

融合的效果：

- 减少中间 tensor 数量
- 降低 temp buffer 峰值
- 减少内存读写带宽
- 往往还能提升执行速度

对于内存紧张平台，融合通常比单纯调度更有效。

---

### **第14页：关键优化五——降低激活精度**

中间激活的精度越低，占用的 temp buffer 越小。

例如：

- FP32：每元素 4 字节
- FP16：每元素 2 字节
- INT8：每元素 1 字节

如果模型和硬件支持，将激活从 FP32 改为 FP16，temp buffer 理论上可接近减半。

这种方法的优势是非常直接，但需要考虑：

- 后端是否支持
- 精度损失是否可接受
- 是否引入额外类型转换开销

---

### **第15页：关键优化六——分块计算**

当某一层输出 tensor 非常大时，可以考虑不一次性生成整个结果，而是按块计算和消费。

例如：

- 按空间 tile 处理 feature map
- 按 token chunk 处理长序列
- 按矩阵分块做乘法

分块计算的本质是：

**用更多调度复杂度或少量额外计算，换取更低的内存峰值。**

这种方法适合极端内存受限设备，但实现复杂度较高。

---

### **第16页：关键优化七——重计算换内存**

某些跨层保留时间很长的张量，占用 temp buffer 很多。  
如果保存它的代价高于重算代价，可以考虑在需要时重新计算，而不是一直保存。

这种策略本质上是：

- 减少长期保留的大 tensor
- 用额外算力换更小的峰值内存

适用于：

- 内存特别紧张
- 算力相对富余
- 某些中间结果重算代价不高

### **第17页：哪些网络结构最容易出现高峰值**

以下模型结构通常会显著增加 temp buffer 压力：

- ResNet 类残差连接
- U-Net 编解码跳连
- DenseNet 密集连接
- Inception 多分支结构
- 大量 Concat 融合路径
- 长序列 Transformer / Attention

这些结构的共同特点是：

- 大张量保留时间长
- 多分支并发存在
- merge 节点较晚出现

### **第18页：工程实现建议**

如果要在静态 shape 推理引擎中落地优化，建议优先实现以下能力：

**1. Tensor 生命周期分析**

记录每个 tensor 的：

- size
- producer
- consumers
- last use

**2. 基于 Arena 的统一内存池**

- 统一管理 temp buffer
- 减少动态分配
- 支持空闲块复用

**3. 基于 last-use 的即时释放**

- 每次消费后更新 use count
- use count 归零立即回收

这三项通常就能解决大部分 temp buffer 过高的问题。

### **第19页：推荐优化优先级**

对于静态 shape 模型，在内存受限设备上的建议优先级是：

1. 生命周期分析 + 内存复用  
2. Arena 静态分配替代动态 malloc/free  
3. 原地执行  
4. 算子融合  
5. 优化执行顺序  
6. 降低激活精度  
7. 分块计算  
8. 重计算换内存

这个顺序的原因是：

- 前几项实现成本相对低，收益通常最大
- 后几项虽然有效，但实现复杂度更高

### **第20页：结论页**

静态 shape 下，ONNX 推理所需的最小 temp buffer，本质上取决于：

**执行过程中所有同时存活的中间张量的峰值内存。**

Skip Connection、分支结构和 Concat 会显著拉长张量生命周期，从而提高峰值内存。

要降低 temp buffer，核心思路是：

- 少保存
- 早释放
- 多复用
- 能原地就原地
- 能融合就融合
- 必要时用计算换内存

对于内存受限设备，最有效的工程路径通常是：

**生命周期分析 + 静态内存规划 + 统一 Arena 复用**
